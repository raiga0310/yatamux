## テスト計画: IPC ハードニング (C-41)

### TC-01: 同一ユーザーが接続できること
- **前提**: Named Pipe の ACL が「同一ユーザーまたは同一ログオンセッション」を許可する設定になっている
- **操作**: yatamux GUI を起動したユーザーと同一ユーザーで `yatamux list-panes` または `ServerConnection::connect()` を実行する
- **期待結果**: Named Pipe 接続が成功し、通常の `ClientMessage` / `ServerMessage` 往復が成立する

### TC-02: 別ユーザー（異なる SID）が接続を試みても拒否されること
- **前提**: テスト端末上で GUI 起動ユーザーとは異なる Windows ユーザーを用意し、SID が異なることを確認する
- **操作**: 別ユーザーで同一の `\\.\pipe\yatamux-<session>` へ接続を試みる
- **期待結果**: 接続は `Access is denied` 相当で拒否され、サーバー側ではセッション操作が開始されない

### TC-03: 認証 handshake 成功（nonce 一致）
- **前提**: IPC 接続直後に nonce 付き handshake request / response を交換する実装がある
- **操作**: サーバーが提示した nonce をそのまま含めて handshake response を返す
- **期待結果**: 接続が認証済み状態へ遷移し、その後の `CreateWorkspace` や `ListPanes` が処理される

### TC-04: 認証 handshake 失敗（nonce 不一致）→ 切断
- **前提**: handshake mismatch を検出したら認証失敗として扱う実装がある
- **操作**: 接続直後に誤った nonce を含む handshake response を送る
- **期待結果**: サーバーは明示的な認証エラーを返したうえで接続を切断し、後続メッセージは受理されない

### TC-05: oversized JSON Lines（例: 1MB 超）を送ると明示エラーで拒否されること
- **前提**: `BufReader.lines()` で読む前段または同等の read loop に 1 メッセージ上限が導入されている
- **操作**: 1MB を超える単一 JSON Lines メッセージを送信する
- **期待結果**: サーバーは「サイズ上限超過」の明示エラーを返して当該クライアントを拒否し、プロセス全体は継続する

### TC-06: 過剰送信クライアントが rate limit / lagged で切断されること
- **前提**: クライアント単位の送信レート制限、または購読遅延時の切断ポリシーが定義されている
- **操作**: 短時間に大量の JSON Lines を送る、または購読メッセージを処理しきれない速度で発生させる
- **期待結果**: サーバーは rate limit 超過または lagged を明示し、対象クライアントだけを切断する

### TC-07: 認証失敗が audit ログに記録されること
- **前提**: 認証失敗・権限不足・入力制限超過を監査ログへ出力する実装がある
- **操作**: nonce 不一致または権限不足のクライアント接続を発生させる
- **期待結果**: ログに失敗種別、接続元識別子、発生時刻などの監査情報が残り、原因追跡ができる

### TC-08: 旧 CLI（認証なし）からの接続を互換モードで許容できること（将来の移行策）
- **前提**: 新旧クライアントを識別し、限定条件付きで旧方式を許容する互換モードがある
- **操作**: 認証 handshake を送らない旧 CLI 相当クライアントで接続し、互換モードを有効にした状態で `ListPanes` を実行する
- **期待結果**: 互換モード有効時のみ接続と要求処理が成功し、無効時は明示エラーで拒否される
